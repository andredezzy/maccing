# MCP codebase reorganization — 2026-06-15

**Goal:** replace the flat 19-file `src/lib/` grab-bag and the 820-line `render-mockup.ts`
god-file with concern-named subsystems and an open/closed renderer. Behavior-preserving —
the MCP tool surface, every tool's output, and all 119 tests stay identical.

## Problems

1. `src/lib/` mixes five unrelated subsystems (transport · id utils · readers · renderer ·
   writers) with no `ls`-level signal of where anything belongs.
2. `render-mockup.ts` (820 LOC) holds the block/view model + text primitives + box
   primitives + ~36 block renderers + ~10 view renderers + 3 entry points, dispatched by a
   buried `switch` — adding a block type means editing a growing switch (open/closed violation).

## Target structure (group by what the code IS — no `lib/`/`utils/`/`shared/`)

```
src/
  server.ts · tool.ts          entry + the ToolModule contract
  tools/                       one file per MCP tool (unchanged)
  notion/                      I/O boundary
    public-client.ts ← notion-public      private-client.ts ← notion-private
    ids.ts           ← normalize-uuid + id-variants + abbreviate-id (merged)
    redact.ts        ← src/redact.ts
  readers/                     raw Notion → agent-friendly TEXT
    page · blocks · markdown · rows · schema · views · object · search · relations
  render/                      the PURE ASCII mockup engine
    index.ts                   public entry: renderPage/renderDatabase/renderBlocks (+ re-exports)
    model.ts                   MockupBlock + ViewBlock unions + interfaces (the render contract)
    text.ts                    displayWidth · clip · padRight · spread · wordWrap
    box.ts                     box · hcat · table-grid · card-fit
    blocks/{index,text,media,layout}.ts   block renderers + type→renderer registry
    views/{index,cards,time,list,data}.ts view renderers + type→renderer registry + dbHeader
    schema.ts          ← render-schema (zod)
    page-model.ts      ← notion-to-page-model      database-model.ts ← notion-to-database-model
  writers/
    upsert-property.ts · reorder-properties.ts
```

## Registry design (the open/closed win)

`render/blocks/index.ts` and `render/views/index.ts` each hold a `Map<type, renderFn>` and a
`renderBlock` / `renderView` dispatcher that looks up the map. Family files (text/media/layout,
cards/time/list/data) define the handlers and register them. Adding a type = a new handler +
one registry entry — never editing a buried switch. Trivial renderers stay grouped by family
(per the "≤10-line helper → inline" rule) rather than one-file-per-type.

## Migration phases (tests + tsc + biome + knip green after EACH)

1. **notion/** — move clients, merge id utils → `ids.ts`, move redact; re-path imports.
2. **readers/** — move the format-* / notion-page / relations family; re-path.
3. **writers/** — move the two write builders; re-path.
4. **render/ (moves)** — move render-schema → `schema.ts`, the two mappers → `*-model.ts`; re-path.
5. **render/ (split)** — extract `render-mockup.ts` → model/text/box/blocks/views/index,
   one piece at a time, tests green between each extraction. `render/index.ts` re-exports the
   public surface so consumers import `../render` (functions) and `../render/model` (types).

Each move uses `git mv` (history preserved) + import re-path; validated by `bun test` (119),
`tsc --noEmit`, `biome check`, `knip`. No version bump — behavior is unchanged, so no
re-activation needed.

# Continued — render-tool unification + exhaustive audit loop (2026-06-15)

After the reorg above, a second arc: unify the renderer around one mental model, then run repeated
multi-auditor audits (each finding adversarially verified) until the findings asymptote to cosmetic
nits. Behavior-preserving throughout; the full suite stayed green at every commit (119 → 177 tests).

## "Everything is a block" — the unifying mental model

Notion's model: a page IS a block, and a page's content is its block children; a database is a block
whose rows are pages (blocks). The renderer now mirrors that:

- The three render tools (`render_page` / `render_database` / `render_blocks`) collapsed into ONE
  **`render_mockup`** whose input is `block | block[]`. `page` and `database` are now block TYPES in the
  `MockupBlock` union (alongside the views and content blocks), each registered in the engine like any
  other block. The block's own `type` is the sole discriminator — no top-level `kind` switch.
- `render/views/` was FOLDED into `render/blocks/` — views (table/gallery/board/calendar/…) are block
  types, so a sibling `views/` dir misrepresented the domain. One `render/blocks/` dir now holds every
  renderer family: content (text · media · layout), containers (page · database), and db views
  (cards · list · time · data · chrome helper).
- `renderPage` and `renderDatabase` both dispatch through the engine registry (`renderRoot` wraps the
  model in its container block) — nothing bypasses the registry.

  notion/    (pure leaf — imports nothing upward)

**Dependency DAG (acyclic, enforced by src/architecture.test.ts):**
`tools → {readers, writers, render, notion}` · `render/blocks → render/engine → render/model` ·
`render → readers` (one-way) · `readers, writers → notion` · `notion` is a pure leaf · `redact → tool`.

A mid-loop note had called the "5-consecutive-zero" target *unreachable* (a stochastic test-tail floor). That
was **wrong**, and instructively so: the interruptions to each zero-streak were never stochastic noise — they
were *genuine finite stragglers* that successive sweeps kept missing because each sweep matched only the
casing/shape it was written for. The recurring offender was **truncations**, closed in three rounds: lowercase
single letters (13), lowercase/camelCase standalone+prefix+suffix (21), then **PascalCase type names (32)**.
The other interruptions were a self-induced cascade from co-locating `fetchViews` (22 → 23 → 27), now fully
resolved. Only once *every* finite category was actually exhausted (not merely believed exhausted) did five
fresh multi-auditor passes in a row return zero. Lesson: a self-grep of *known* categories can't find *unknown*
ones — only a fresh adversarial multi-auditor can — and "exhausted" means grep-verified across every casing,
not asserted.
