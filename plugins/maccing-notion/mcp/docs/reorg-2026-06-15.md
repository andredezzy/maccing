# MCP codebase reorganization ÔÇö 2026-06-15

**Goal:** replace the flat 19-file `src/lib/` grab-bag and the 820-line `render-mockup.ts`
god-file with concern-named subsystems and an open/closed renderer. Behavior-preserving ÔÇö
the MCP tool surface, every tool's output, and all 119 tests stay identical.

## Problems

1. `src/lib/` mixes five unrelated subsystems (transport ┬Ě id utils ┬Ě readers ┬Ě renderer ┬Ě
   writers) with no `ls`-level signal of where anything belongs.
2. `render-mockup.ts` (820 LOC) holds the block/view model + text primitives + box
   primitives + ~36 block renderers + ~10 view renderers + 3 entry points, dispatched by a
   buried `switch` ÔÇö adding a block type means editing a growing switch (open/closed violation).

## Target structure (group by what the code IS ÔÇö no `lib/`/`utils/`/`shared/`)


## Registry design (the open/closed win)

`render/blocks/index.ts` and `render/views/index.ts` each hold a `Map<type, renderFn>` and a
`renderBlock` / `renderView` dispatcher that looks up the map. Family files (text/media/layout,
cards/time/list/data) define the handlers and register them. Adding a type = a new handler +
one registry entry ÔÇö never editing a buried switch. Trivial renderers stay grouped by family
(per the "ÔëĄ10-line helper Ôćĺ inline" rule) rather than one-file-per-type.

## Migration phases (tests + tsc + biome + knip green after EACH)

1. **notion/** ÔÇö move clients, merge id utils Ôćĺ `ids.ts`, move redact; re-path imports.
2. **readers/** ÔÇö move the format-* / notion-page / relations family; re-path.
3. **writers/** ÔÇö move the two write builders; re-path.
4. **render/ (moves)** ÔÇö move render-schema Ôćĺ `schema.ts`, the two mappers Ôćĺ `*-model.ts`; re-path.
5. **render/ (split)** ÔÇö extract `render-mockup.ts` Ôćĺ model/text/box/blocks/views/index,
   one piece at a time, tests green between each extraction. `render/index.ts` re-exports the
   public surface so consumers import `../render` (functions) and `../render/model` (types).

Each move uses `git mv` (history preserved) + import re-path; validated by `bun test` (119),
`tsc --noEmit`, `biome check`, `knip`. No version bump ÔÇö behavior is unchanged, so no
re-activation needed.

# Continued ÔÇö render-tool unification + exhaustive audit loop (2026-06-15)

After the reorg above, a second arc: unify the renderer around one mental model, then run repeated
multi-auditor audits (each finding adversarially verified) until the findings asymptote to cosmetic
nits. Behavior-preserving throughout; the full suite stayed green at every commit (119 Ôćĺ 177 tests).

## "Everything is a block" ÔÇö the unifying mental model

Notion's model: a page IS a block, and a page's content is its block children; a database is a block
whose rows are pages (blocks). The renderer now mirrors that:

- The three render tools (`render_page` / `render_database` / `render_blocks`) collapsed into ONE
  **`render_mockup`** whose input is `block | block[]`. `page` and `database` are now block TYPES in the
  `MockupBlock` union (alongside the views and content blocks), each registered in the engine like any
  other block. The block's own `type` is the sole discriminator ÔÇö no top-level `kind` switch.
- `render/views/` was FOLDED into `render/blocks/` ÔÇö views (table/gallery/board/calendar/ÔÇŽ) are block
  types, so a sibling `views/` dir misrepresented the domain. One `render/blocks/` dir now holds every
  renderer family: content (text ┬Ě media ┬Ě layout), containers (page ┬Ě database), and db views
  (cards ┬Ě list ┬Ě time ┬Ě data ┬Ě chrome helper).
- `renderPage` and `renderDatabase` both dispatch through the engine registry (`renderRoot` wraps the
  model in its container block) ÔÇö nothing bypasses the registry.
